# План разработки Mission Recorder

## 1. Краткое понимание продукта

Mission Recorder - это локальный кроссплатформенный рекордер экрана "как Bandicam", но бесплатный, открытый и реализованный в Kotlin/KMP-стиле. Продукт должен поддерживать запись всего экрана, выбранного монитора, выделенной области, конкретного окна или приложения, микрофона и системного звука. Дополнительно нужны replay buffer последних `N` минут, экспорт кадров, CLI для автоматизации и Compose Multiplatform GUI для удобного управления.

Функции уровня Bandicam должны быть реализованы в стиле Mission Recorder:

- без обязательного облака;
- без скрытого захвата и неявных разрешений;
- без платных ограничений на базовые сценарии;
- с локальными файлами, понятными настройками и открытой архитектурой;
- с явным разделением общей логики и platform-specific API.

## 2. Аудит текущего репозитория

- главное и always-on-top mini-окно используют фирменную иконку; native distributions получают `ICO` для Windows, `ICNS` для macOS и `PNG` для Linux;
- GUI output поддерживает ручной путь, явный native save dialog и редактор profile directory/fileNamePattern с `{timestamp}`/`{profile}`;
- GUI Output открывает родительский каталог текущего output path через desktop adapter и создаёт его при отсутствии;
- `export-frames` читает MP4 и поддерживает `--layout separate|sheet`;
- `:app` подключен к CLI runner и desktop recording backend с AWT video и явным выбором Java Sound microphone;
- CLI/GUI на Windows обнаруживают все active render endpoints с понятными Windows friendly names, поддерживают явный выбор, system-only и mixed mic + system AAC;
- app-level тест проходит через `DesktopRecordingCommandBackend`, `RecordingController` и frame-sequence encoder на fake video/audio frames без реального захвата устройств.
- app-level тест проходит через `FrameSequenceExportFramesCommandBackend` и экспортирует кадры из synthetic `.mrec`.

Что не реализовано:

- desktop pipeline пишет MP4; screen/monitor/region основаны на AWT, window/application - на User32/GDI для Windows и Xlib для Linux/X11, microphone - на Java Sound;
- graceful Ctrl+C готов для `record` и бессрочного foreground `replay run`; `--run-for` остался опциональным automation deadline;
- versioned локальный CLI control channel готов для обычной записи и foreground replay: replay поддерживает status, отдельный snapshot save без остановки и финальный save-and-stop;
- `capture-core`, `audio-core`, `capture-platform-api`, `replay-buffer`, common schema `settings`, `hotkey-core` и `compose-ui` используют KMP source sets; CLI, encoder, file store и desktop orchestration остаются JVM-specific, полное target-покрытие еще не завершено;
- system audio доступен на Windows через WASAPI и на Linux через PulseAudio/PipeWire monitor sources при наличии `pactl`/`parec`; выбор endpoint готов, app/PID filtering не реализован;
- нет Windows Graphics Capture/DXGI, Wayland portal, ScreenCaptureKit replacement для CoreGraphics fallback и Android adapters;
- settings module подключен к CLI `settings init/validate/show` и `record profile`; GUI умеет выбирать, создавать, атомарно сохранять и удалять именованные profile snapshots с source/audio/video/encoder/replay/output; directory/fileNamePattern редактируются в GUI, скрытые codec/container поля - через JSON;
- после каждой завершенной GUI-записи следующий output path заново вычисляется из pattern активного профиля, а завершенный MP4 остается отдельным storyboard input;
- always-on-top мини-панель управления, runtime mute и локальное восстановление положения готовы; desktop visual smoke на целевых window managers еще не завершен;
- Windows global hotkeys готовы через сохраняемый opt-in menu toggle; adapters для macOS/Linux/Android и редактирование комбинаций еще не реализованы;
- replay buffer подключен к GUI и foreground CLI, а save-replay hotkey - к Windows GUI; daemon еще не реализован;
- экспорт из WebM пока не заявлен и не проверен; production-запись использует MP4;
- permissions flow задокументирован в `docs/permissions.md`; Linux JVM gateway различает X11 и fail-closed Wayland/unknown session, а ownership/thread-affinity/cleanup нативных ресурсов зафиксированы в [docs/native-ownership.md](docs/native-ownership.md).

Решенная техническая проблема:

- Gradle запускается на Java 21, поэтому `buildSrc` и JVM-модули закреплены на toolchain 21. `.\gradlew.bat check` проходит локально.

## 3. Предлагаемая архитектура

Целевая модульная структура:

- `:capture-core` - платформонезависимая доменная логика записи, модели, state machine, use cases, ошибки, события, Flow-контракты.
- `:capture-platform-api` - KMP common интерфейсы платформенных адаптеров: source discovery, video capture, permissions, window/application metadata.
- `:capture-windows-jna` - текущий Windows adapter для discovery и window/app capture через User32/GDI.
- `:capture-linux-x11` - текущий Linux/X11 adapter для EWMH discovery и window/app capture через Xlib.
- `:capture-windows-wgc` - будущий Windows Graphics Capture/DXGI adapter для robust screen/window/app capture.
- `:capture-macos-coregraphics` - текущий macOS window/app adapter и native permissions; `:capture-macos-screencapturekit` - целевая замена deprecated image API.
- `:capture-linux` - Linux adapter для PipeWire, xdg-desktop-portal и X11 fallback.
- `:capture-android` - Android adapter для MediaProjection.
- `:audio-core` - KMP common модели аудио, микширование, runtime mute, уровни, sync metadata и routing.
- `:audio-platform-api` - интерфейсы микрофона, system audio, app audio filtering.
- `:audio-windows-wasapi` - Windows loopback adapter; `:audio-linux-pulse` - Linux PulseAudio/PipeWire monitor adapter; `:audio-macos` и `:audio-android` - будущие platform audio adapters.
- `:encoder` - интерфейсы encoder/muxer, настройки кодеков, timestamp handling, output writer.
- `:settings` - KMP schema/validation профилей и JVM atomic JSON migration/store.
- `:replay-buffer` - rolling buffer последних `N` минут, политика памяти/диска, save replay.
- `:export` - экспорт кадров в изображения, naming pattern, contact sheet при необходимости.
- `:hotkey-core` - KMP-модель bindings/actions/events глобальных горячих клавиш; **готово**
- `:hotkey-windows-jna` - Win32 `RegisterHotKey` adapter с отдельным message-loop thread; **готово**
- `:cli` - CLI-команды, parsing, exit codes, machine-readable output.
- `:compose-ui` - stateless Compose screens, view models, UI state.
- `:app-desktop` - desktop wiring для CLI/GUI на JVM.
- `:app-android` - Android app wiring, permissions flow, foreground service.

Границы ответственности:

- core-модули не знают про JNI, WinAPI, ScreenCaptureKit, PipeWire, MediaProjection, file dialogs или UI;
- platform adapters реализуют только низкоуровневый доступ к устройствам и ОС;
- encoder не принимает UI-модели и не знает про Compose;
- CLI и GUI используют одни и те же use cases;
- production dispatchers и scopes связываются только на application boundary;
- тесты доменной логики используют fake adapters.

## 4. Пайплайн записи

Основной pipeline:

1. Пользователь выбирает `CaptureSource`.
2. Settings validation проверяет источник, регион, аудио, output path, codec и разрешения.
3. Permission service запрашивает или проверяет права платформы.
4. Video adapter создает поток `VideoFrame`.
5. Audio adapters создают потоки `AudioFrame` для микрофона и системного звука.
6. Synchronizer выравнивает аудио и видео по единому clock abstraction.
7. Encoder кодирует video/audio frames.
8. Muxer пишет контейнер в temporary output.
9. После успешного stop файл атомарно перемещается в целевой output.
10. Metrics и final state публикуются через `StateFlow`.

Видео:

- frame timestamp должен быть monotonic;
- нужно хранить размер, pixel format, stride, source id, scale factor;
- dropped frames должны попадать в metrics;
- region/window/app capture должен нормализовать координаты с учетом DPI/scale.

Микрофон:

- выбирать устройство по stable id;
- поддерживать sample rate, channel count, format;
- публиковать уровень сигнала для GUI;
- корректно освобождать device handle при cancel/failure.

Системный звук:

- Windows: WASAPI loopback;
- macOS: ScreenCaptureKit audio там, где применимо, или CoreAudio route;
- Linux: PipeWire/PulseAudio monitor sources;
- Android: AudioPlaybackCapture API для разрешенных приложений.

Синхронизация:

- единый `MediaClock`;
- timestamp normalization на входе adapter слоя;
- drift correction для длинных записей; **готов bounded KMP corrector перед mixing; многочасовой hardware smoke остается**
- explicit policy для dropped/duplicated frames.

Ошибки и отмена:

- все long-running операции должны быть cancellable;
- cleanup выполняется в `finally`;
- ошибки маппятся в доменные типы: permission denied, source unavailable, encoder failed, disk full, device lost;
- state machine не должна зависать в `Stopping` или `Preparing`.

## 6. Платформенная стратегия

### Windows

Видео:

- AWT screen/monitor/region и JNA User32/GDI window/app prototype реализованы;
- следующим native video backend-ом использовать Windows Graphics Capture или DXGI Desktop Duplication;
- для window/app capture использовать Windows Graphics Capture при доступности;
- учитывать DPI awareness и multi-monitor.

Аудио:

- WASAPI loopback для выбранного active render endpoint реализован; Java Sound остается microphone adapter-ом;
- mixed microphone + loopback нормализуется в общий 48 kHz stereo поток;
- app-specific audio filtering рассматривать отдельно через доступные Windows APIs и ограничения ОС.

Разрешения и риски:

- права на экран обычно проще, чем на macOS/Android;
- возможны проблемы с protected content и UAC/elevated windows;
- hardware encoding зависит от GPU и драйверов.

Реализовать первым:

- Windows full screen или monitor video-only MVP.

### macOS

Видео:

- ScreenCaptureKit для screen/window/app capture;
- учитывать Retina scale и permission model.

Аудио:

- ScreenCaptureKit audio для system audio там, где применимо;
- AVAudioEngine/CoreAudio для микрофона.

Разрешения и риски:

- нужны Screen Recording и Microphone permissions;
- пользователь должен явно разрешить доступ в System Settings;
- app capture и audio capture зависят от версии macOS.

Реализовать после Windows MVP:

- screen/window capture с понятным permissions flow.

### Linux Ubuntu

Видео:

- PipeWire + xdg-desktop-portal для Wayland;
- X11 fallback через XCB/FFmpeg/GStreamer при необходимости.

Аудио:

- PipeWire или PulseAudio monitor sources;
- app audio filtering возможен не во всех конфигурациях.

Разрешения и риски:

- разные desktop environments ведут себя по-разному;
- Wayland требует portal flow;
- window capture может быть ограничен compositor-ом.

Реализовать после стабилизации domain/core:

- PipeWire screen capture через portal.

### Android

Видео:

- MediaProjection API;
- foreground service для длительной записи;
- MediaCodec/MediaMuxer для encoding.

Аудио:

- AudioRecord для микрофона;
- AudioPlaybackCapture API для разрешенных приложений и системных ограничений.

Разрешения и риски:

- нужны runtime permissions;
- background ограничения;
- нельзя гарантировать захват звука всех приложений;
- UI/UX permissions flow критичен.

Реализовать отдельным этапом:

- Android app после desktop core и encoder abstractions.

## 11. Риски и неизвестные

Системный звук:

- API сильно отличаются между Windows, macOS, Linux и Android;
- на Android нельзя гарантировать захват всех приложений;
- на Linux поведение зависит от PipeWire/PulseAudio и desktop environment.

Фильтрация по приложениям:

- не везде доступна надежно;
- может потребовать платформенных fallback-стратегий;
- UI должен честно показывать unsupported state.

Захват окон/приложений:

- protected content может быть недоступен;
- elevated/admin windows на Windows могут вести себя иначе;
- Wayland может ограничивать прямой доступ.

Разрешения:

- macOS требует явного Screen Recording permission;
- Android требует MediaProjection consent и foreground service;
- Linux portal flow зависит от окружения.

Производительность:

- software encoding может перегружать CPU;
- hardware encoding зависит от драйверов;
- multi-monitor/high-DPI/60fps требуют раннего profiling.

Синхронизация:

- audio/video drift на длинных записях;
- dropped frames;
- разные clock sources в platform adapters.

## 13. Рекомендуемый порядок первых задач

1. Решить JDK/toolchain mismatch: JDK 25 для Gradle runtime или согласованное снижение toolchain.
2. Добиться прохождения `.\gradlew.bat check`.
3. Зафиксировать решение по переходу от JVM-only к KMP/Compose Multiplatform.
4. Создать `:capture-core`.
5. Перенести демонстрационный `utils` за пределы продуктовой логики или удалить позже отдельной задачей.
6. Описать доменные модели записи.
7. Реализовать `RecordingState` и state machine.
8. Добавить fake adapters для видео и аудио.
9. Написать tests на start, stop, cancel, failure.
10. Спроектировать `VideoFrame`, `AudioFrame`, `MediaTimestamp`.
11. Создать `:settings` с validation и serialization. **готово**
12. Создать `:encoder` с fake encoder и контрактом muxer/output writer.
13. Создать `:cli` skeleton и `list-sources`.
14. Реализовать первый Windows video-only adapter prototype.
15. Подключить запись output файла через encoder/muxer.
16. Добавить CLI `record screen`.
17. После CLI MVP подключить Compose Desktop shell.
18. Добавить выбор монитора и region capture.
19. Добавить microphone capture. **частично готово: Java Sound PCM -> bounded drift correction -> AAC в MP4; native platform adapters и многочасовой hardware smoke остаются**
20. После стабилизации audio/video sync переходить к system audio, replay buffer и app capture. **Windows WASAPI и Linux PulseAudio/PipeWire system audio, replay buffer, global hotkeys и User32/GDI app/window prototype готовы; daemon, app-audio filtering, macOS и Android audio остаются**
21. Добавить opt-in always-on-top мини-панель с общим recording state, record/stop и runtime mute для microphone/system audio. **готово; desktop visual smoke остается**
